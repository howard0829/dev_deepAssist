01 · Overview

개요와 목표

DeepAssist는 "Claude Agent SDK의 query() 옵션을 조립하고, 워크스페이스 도구 위임 브리지를 운영하는 서버측 오케스트레이션 계층"이다. 에이전트 루프 자체는 구현하지 않는다.

  • Orchestrator = DeepAssist — MCP 서버·RAG·skills를 관리해 core agent에 넘긴다.
  • Core agent = Claude Agent SDK(Python). 서버에 상주.
  • UI = 기존 VSCode 확장 그대로 — 기존 WebSocket 위임 프로토콜 유지.
  • LLM = LiteLLM 프록시 → vLLM·OpenAI — 원격 공유 서버 상주.
  • 범위 — 파일/셸만 위임한다. LSP는 제외.

UI 재사용 범위기존 위임 프로토콜 그대로 유지위임 토폴로지와 정합. protocol.py + tool-executor.ts 재사용
워크스페이스 도구내장 Bash/Read/Write/Edit/Glob/Grep 비활성화 → 위임형 MCP 도구로 대체훅으로는 "다른 곳에서 실행" 불가(내장이 서버에서 먼저 실행됨)

04 · Components

구성요소

Client · TS

VSCode 확장

대부분 재사용. 서버 메시지가 Agent SDK 이벤트 기반으로 바뀌므로 표시 계층 매핑만 조정.

  • webview UI(chat · settings)
  • ws-client — 기존 프로토콜
  • tool-executor — 파일·Bash native 실행, dangerous 체크, 변경전 캐시, 되돌리기
Server · Python

DeepAssist 오케스트레이터

이 프로젝트의 핵심 신규 계층.

  • 세션별 ClaudeAgentOptions 조립
  • 위임형 MCP 도구를 세션 bridge에 바인딩
  • 서버직접 MCP(RAG) 등록
  • skills · subagents · system prompt · permission 주입
  • SDK 메시지 → UI 프로토콜 번역
Server · Python

Claude Agent SDK

에이전트 루프·컨텍스트 관리·도구 호출. 서버 프로세스에서 구동. ANTHROPIC_BASE_URL을 로컬 LiteLLM으로 지정.

Server · reuse

RAG · MCP

기존 g_tas_server/rag·mcp 자산을 서버직접 MCP 도구로 래핑해 재활용. 무거운 인덱싱은 서버 유지.

LLM Backend

LiteLLM → vLLM · OpenAI

Anthropic 포맷 /v1/messages를 수신해 vLLM/OpenAI로 변환. cache_control·thinking·count_tokens 처리 필요(§11).

05 · Tooling

도구 아키텍처 · 위임

기존 G-TAS의 execute=callable(서버직접) / None(클라위임) 구도를 계승한다. 모든 도구를 SDK MCP 도구로 정의하되, handler 내부에서 위임 여부가 갈린다.

분류도구실행 위치구현
클라 위임bash · read · write · edit · glob · grep클라 워크스페이스handler가 bridge.request_and_wait로 위임
서버 직접RAG/knowledge 검색 · scratch · 거대파일 변환 · ask_user서버handler가 서버 리소스 직접 호출
비활성화SDK 내장 Bash/Read/Write/Edit/Glob/Grep—disallowed_tools로 제거 (서버 FS 오염 방지)
제외LSP 계열—이번 범위 밖

위임 메커니즘

내장 도구는 서버 FS에서 실행되므로 반드시 끄고, 동일 책임의 위임형 MCP 도구로 대체한다. 훅(PreToolUse/PostToolUse)은 내장 도구가 서버에서 먼저 실행되는 문제로 위임 용도에 부적합.

06 · Lifecycle

요청 라이프사이클

  1. 사용자 프롬프트클라 → 서버 : user_message(prompt, workspace, provider_config …)
  2. 세션 준비DeepAssist가 ClaudeAgentOptions 조립 — 위임 MCP + RAG MCP + skills + system_prompt
  3. 에이전트 실행for msg in query(prompt, options) — StreamEvent→agent_text, tool_use→tool_call_update, 위임 도구는 tool_request↔tool_result 왕복, 파일수정→modified_file_diff, 위험작업→approval_request
  4. 완료ResultMessage → agent_complete(response, modified_files, diffs, metrics)

08 · Options

query() 옵션 구성

options = ClaudeAgentOptions(
    model=provider_config.model,                 # per-query 모델
    system_prompt={                              # 프리셋 + 위임 도구 지침 (§9.1)
        "type": "preset", "preset": "claude_code",
        "append": DEEPASSIST_TOOL_GUIDE,
    },
    disallowed_tools=[                           # 내장 워크스페이스 도구 제거
        "Bash","Read","Write","Edit","Glob","Grep",
    ],
    mcp_servers={                                # 위임형 + 서버직접
        "deepassist": create_sdk_mcp_server(
            name="deepassist",
            tools=[bash_tool, read_tool, write_tool,
                   edit_tool, glob_tool, grep_tool],  # handler가 bridge 위임
        ),
        "knowledge": create_sdk_mcp_server(
            name="knowledge", tools=[rag_search, ...],  # 서버직접
        ),
    },
    allowed_tools=["mcp__deepassist__bash", ...,
                   "mcp__knowledge__rag_search"],  # 사전 승인
    permission_mode="default",                   # canUseTool 승인 게이트
    agents={...},                                # subagents (선택)
    skills="all",                              # skills 로드
    include_partial_messages=True,             # 스트리밍 UI
    cwd=workspace_label,                         # 명목상(실 실행은 클라)
    env={"ANTHROPIC_BASE_URL": litellm_url,
         "ANTHROPIC_AUTH_TOKEN": token},
)

bash_tool 등은 @tool로 정의하고, handler가 세션 bridge를 캡처해 tool_request를 위임한다.

09 · Conflicts

충돌 처리

9.1 · 도구 이름 · 시스템 프롬프트 정합성 — 유일한 튜닝 지점

MCP 도구는 항상 mcp__<server>__<name> 프리픽스라 이름이 Bash/Edit가 될 수 없다. claude_code 프리셋은 내장 도구명을 지칭하므로 어긋난다. 처리: 프리셋 + append로 위임 도구의 이름·용도·규칙을 명시(필요 시 커스텀 프롬프트). → 구현 초기 프롬프트 스파이크 대상.

9.2 · Edit / Read 의미론 재현

내장 Edit의 "Read 후 수정, staleness 검사" 규약을 위임 도구가 스스로 일관되게 구현. 기존 클라 _beforeContentCache·diff·user_reject_change 로직 재활용.

9.3 · 지연 누적

도구 호출마다 서버↔클라 왕복. Agent SDK는 도구 호출이 잦아 지연이 쌓인다(G-TAS가 이미 감수하던 특성). 완화: tool_progress heartbeat, 병렬 안전 도구는 병렬 실행 검토.

11 · Verify

검증 · 리스크

  • LiteLLM /v1/messages 호환 — Agent SDK가 보내는 cache_control·thinking·count_tokens를 vLLM/OpenAI로 정상 변환/패스스루하는지. → 최소 왕복 스파이크.
  • 위임 MCP handler ↔ bridge 연동 — async handler에서 bridge.request_and_wait await가 세션 스코프로 정확히 매칭되는지.
  • 내장 도구 비활성화 후 도구 선택 품질 — 위임형 MCP 도구만 남겼을 때 모델이 적절히 사용하는지(§9.1 프롬프트 튜닝과 함께).
  • 모델 능력 — vLLM 로컬 모델의 도구 호출·장기 에이전트 스태미나(interleaved thinking 보존 등 고려).
